# ADR-003: MVP段階はAPI認証なしで運用する

## Status

Accepted

## Context

Go API サーバー（ルートの PostgreSQL 版・`backend/` のインメモリ版の双方）の全エンドポイントには認証・認可が実装されていない。誰でも `GET/POST /api/...` を呼び出せる状態である。

これは [ADR-002](002-mvp-fixed-user-id.md)（固定ユーザーID運用）と対をなす意図的な MVP 設計で、現段階の優先事項は射形分析・稽古記録・道場運営ダッシュボードといったコア機能の UX 検証にある。認証基盤（IdP 連携・トークン検証・セッション管理）の構築はフロント／バックエンド双方に大きな変更を要するため、本番リリース前フェーズに先送りする。

本 ADR は「認証がないこと」が見落としではなく明示的な決定であることを記録し、本番運用に移る際の必須対応を定義するために作成する（Issue #27）。

## Decision

MVP 段階では API に認証を実装せず、以下の制約の下で運用する。

1. **ローカル／デモ用途に限定する。** 本番環境（公開インターネット）へはデプロイしない。
2. **秘匿・個人情報を保存しない。** シードデータおよびデモ入力はダミーに限定する。
3. **CORS を許可オリジンリストで制限する。** `CORS_ORIGIN`（カンマ区切りで複数指定可）で許可オリジンを明示し、ワイルドカードは使わない（Issue #31 で対応済み）。
4. **API 設計は認証導入を織り込む。** `userId` はリクエストパラメータで受け取る方式を維持し、認証導入時にミドルウェアからの注入へ差し替え可能にしておく。

## Consequences

### Positive

- 認証基盤の構築を後回しにでき、コア機能の開発・検証に集中できる。
- CORS 制限により、想定外オリジンからのブラウザ経由アクセスは抑止される。

### Negative

- **本番環境での運用は不可**（無認証で全データが読み書き可能なため）。
- 認可（所有者チェック等）が無く、任意の `userId` を指定して他ユーザーのデータを操作できてしまう。

### 本番リリース前の必須対応（Definition of Done）

1. **認証**: Firebase Auth / Google OIDC 等で ID トークンを発行し、Go API に JWT 検証ミドルウェアを追加する。
2. **認可**: 認証済みユーザーのサブジェクトから `userId` を導出し、リクエストの `userId` パラメータ依存を廃止。リソース所有者チェックを追加する。
3. **保護範囲**: `GET /api/health` を除く全エンドポイントを認証必須にする。MVP で対象外にした事実は本 ADR に記録済み（`~/.claude/rules/security.md` の「認証が必要なエンドポイントには認証を付ける。MVPで対象外ならADRに明記」に準拠）。
4. **監査**: 認証導入後、無認証で到達可能なエンドポイントが残っていないことを E2E で検証する。
