# ADR-001: MVP ではフロントエンドに集中し、バックエンドはモック

## Status

Accepted

## Context

kyudo-dojo-hub は弓道の稽古記録・射形分析・道場運営を統合するプラットフォームである。技術スタックは TypeScript/React（フロントエンド）、Go gRPC + REST gateway（API層）、Python MediaPipe（骨格推定）の3層構成を予定している。

MVP段階で3つの技術スタックを同時に立ち上げると、以下のリスクがある:
- 開発リソースの分散によるデリバリー遅延
- API設計が固まる前にバックエンド実装が進むことによる手戻り
- インフラ構築コストが初期段階で発生

## Decision

MVP ではフロントエンド（TypeScript/React）に集中し、Go API と Python ML はモック API 層で代替する。

具体的には:
1. `src/lib/mock-api.ts` にインメモリのモック API を実装
2. データモデルは `src/types/` に TypeScript 型として定義
3. モック API は本番 API と同じインターフェースを持ち、将来の差し替えを容易にする
4. サンプルデータ（10件以上）を含め、UI開発・テストに十分なデータを提供

## Consequences

### Positive
- フロントエンドのUX検証を高速に回せる
- データモデルの型定義がAPIコントラクトの設計文書として機能する
- モック→本番の差し替えポイントが明確

### Negative
- モック API と本番 API の乖離リスク（定期的な型チェックで軽減）
- ネットワーク遅延・エラーハンドリングの検証が不十分になる可能性
- モックデータのメンテナンスコスト

### Mitigation
- モック API に意図的な遅延（50-200ms）を追加し、非同期UIの動作確認を可能にする
- エラーケースもモックに含める（404、バリデーションエラー等）
