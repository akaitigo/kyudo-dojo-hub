# ADR-002: MVP段階は固定ユーザーIDで運用する

## Status

Accepted

## Context

kyudo-dojo-hub のフロントエンドでは `CURRENT_USER_ID = "user-001"` がハードコードされている（`src/pages/VideoAnalysisPage.tsx` 他）。MVP段階では認証基盤が未構築であり、ユーザー認証の導入はフロントエンド・バックエンド双方に大きな変更を要する。

現時点の優先事項は射形分析・稽古記録のコア機能のUX検証であり、認証は本番リリース前に対応する。

## Decision

MVP段階では固定ユーザーID（`user-001`）をフロントエンドにハードコードして運用する。

具体的な制約:
1. フロントエンドの `CURRENT_USER_ID` 定数で固定ユーザーを使用
2. バックエンドのAPIは `userId` をリクエストパラメータとして受け取る設計を維持（認証導入時にミドルウェアから注入に変更可能）
3. シードデータに複数ユーザーを含め、管理画面等のマルチユーザーUIは開発可能にする

## Consequences

### Positive
- 認証基盤の構築を後回しにでき、コア機能の開発に集中できる
- API設計は `userId` パラメータ方式なので、認証導入時の変更が限定的

### Negative
- セキュリティ上、本番環境での運用は不可
- マルチユーザーシナリオのE2Eテストが限定的

### Next Steps
- 本番リリース前にFirebase Auth / Google OIDCによる認証基盤を導入
- `CURRENT_USER_ID` を認証セッションから取得するように変更
- バックエンドにJWT検証ミドルウェアを追加
